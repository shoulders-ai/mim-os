# Known Issues

Logged items that require a separate effort. Fix small things inline; log larger ones here.

## Open

- **Animated agent titles cause per-frame persistence and renderer events.** `agentSessions.ts#onData` persists every changed `titleHint`; Braille spinner frames and Codex's blinking Action Required marker therefore rewrite the session record and emit `session.status` even when the semantic status and stable title text are unchanged. Fix direction: separate the raw title used for protocol detection from a normalized, stable display/persistence hint, while retaining title-text changes that drive auto-titling. This needs focused lifecycle and auto-title regression coverage rather than being folded into a status-classification patch.
- **`package.tools.execute` approval prompt masks the target tool's policy — un-named tools only.** Named granted package tools now dispatch directly under their public name (`createAiSdkTools` checks the ToolRegistry), so the gate evaluates their own dynamic policy and keys session grants on the real name. Un-named chat-audience tools still route through `package.tools.execute` (`general`/`low` → one generic "This changes your workspace" prompt) and share the `sessionId:package.tools.execute` always-allow key, so allowing one benign un-named call still grants every other un-named package tool for the session. Fix direction: when the gated tool is `package.tools.execute`, resolve the target from the `name` param and evaluate the target's dynamic policy (via `getDynamicToolPolicy`), and key session grants on the target name. `src/main/security/gate.ts`, `src/main/ai/aiRuntime.ts` (`createAiSdkTools`). Note: dispatch-level protection for non-chat-audience tools now exists in `packageRuntime.ts#executeTool` (refused for `ai` callers), so this issue is about prompt fidelity and grant granularity, not reachability of ui-only tools.
- **Collision-diagnosed named tools dispatch to the colliding registration.** When a package declares a named tool whose public name collides with an existing registration (core tool or an earlier package), `namedPackageTools.sync` refuses the registration and emits a diagnostic — but `listChatTools` still catalogs the tool under the declared name, so the chat dynamic tool's direct-by-name dispatch reaches the *other* owner's registration (under that tool's own gate policy) instead of the package's shadowed implementation. The app is already flagged as misconfigured, and static SDK keys still shadow dynamic ones, but the clean fix is to exclude collision-refused tools from the chat catalog (thread `named` + registration outcome through `toolSummary`) or dispatch directly only for names the named-tool sync owns (`ownedNames`). `src/main/packages/namedPackageTools.ts`, `src/main/tools/packageRuntime.ts` (`toolSummary`), `src/main/ai/aiRuntime.ts` (`createAiSdkTools`).
- **Scoped package agents cannot allowlist their own un-named tools.** `agentMounts` accepts an agent `tools` entry that names the app's own un-named tool (mechanical `pkg_<hash>__<id>` public name, `isOwnTool` check), but such tools have no ToolRegistry registration, so chat dispatch falls back to `package.tools.execute`, which the gate denies as outside the delegated surface. Named granted tools work. Fix direction: either register all chat-audience tools of apps that declare agents, or reject un-named tool ids in agent allowlists at mount time with a diagnostic telling the author to declare `name` + `provides.tools`. Do not add `package.tools.execute` to allowlists — that would grant every package's tools. `src/main/ai/agentMounts.ts`, `src/main/ai/aiRuntime.ts`, `src/main/packages/namedPackageTools.ts`.
- **Claude Sonnet 5 introductory API pricing expires after August 31, 2026.** The current catalog correctly uses the temporary $2 input / $10 output rates (plus $2.50 five-minute cache writes and $0.20 cache reads), but Anthropic's published standard rates begin September 1: $3 input, $15 output, $3.75 five-minute cache writes, and $0.30 cache reads per million tokens. Update `resources/ai-models.json` on that date, or add effective-dated catalog pricing before then so estimates switch automatically.
- **Policy-disabled tools are invisible instead of discoverable.** When a Settings > Tools group is off, the tool is silently omitted from the AI toolset and the MCP list, so agents conclude the capability doesn't exist and route around it (observed: agent reconstructed a calendar agenda from Gmail invites while `calendar.events` was one toggle away — connected, scopes granted, group `defaultEnabled: false`). Agents should be able to see "exists but disabled — ask the user to enable Read Calendar in Settings > Tools" rather than nothing. Fix direction: expose disabled-but-available groups in the system prompt tool-set section and/or a `connections_status`-style discovery result, without exposing the tools themselves. `src/main/tools/toolPolicy.ts`, `src/main/ai/systemPrompt.ts`, `src/main/index.ts` (`getNamedMcpTools`).
- **Draft vocabulary leaked into two docs ahead of the code.** `docs/workbench-navigation.md` and `docs/observability.md` call the fourth sidebar surface "Review", but the shipped UI label is "Monitor" (`ShellSidebar.vue`). Since the ontology/vocabulary migration is drafted but not approved, either revert the docs to shipped terms or execute the migration — do not leave the layers disagreeing. The user manual (`manual/`) follows the shipped UI.
- **Connections settings copy mixes "websites" and "domains".** `ConnectionsSettingsPanel.vue` has group titles "Website Access" and "Domains" and a caption "granted websites" for the same concept. Pick one term.
- **mim-apps registry contains duplicate app `word-count-2`** — identical name-and-description copy of `word-count` in `index.json`. Delete from the registry (external repo).
- **Electron is behind on security patches (`^35.0.0`).** A major-version upgrade is needed — it rebuilds all native modules (better-sqlite3, node-pty, keytar) and may touch BrowserWindow/security APIs. Do as a focused upgrade with a full app smoke test.
- **`keytar.node` arch drift after `electron-rebuild`.** If test collection failures appear across files importing keytar, rebuild keytar for the host arch or mock it at the test boundary.
- **No single-instance or workspace lock.** Two app instances can open the same workspace and corrupt `.mim/` state. Needs a lockfile or IPC-based single-instance guard.
- **Workspace deletion or unmount while open is never detected.** The app continues operating against a missing path. No recovery or re-prompt path exists.
- **Hung job blocks its non-parallel slot until restart.** A package job that never resolves holds the concurrency slot forever. No timeout or watchdog mechanism exists.
- **Linux: keytar throws when no OS keyring is running.** On minimal WMs or headless Linux without GNOME Keyring/KWallet, secret-backed integrations fail loudly; needs an encrypted local fallback.
- **Platform shell hardcodes slides-specific result fields.** `PackageRunView` and related components contain slides-specific assumptions that should be generic package capabilities.
- **`web.read`: Readability crash fallback has no test.** The try/catch around Readability.parse() (added after `shoulde.rs` crashed it) has no test exercising the catch path.
- **Apps settings tests are out of sync with the current Apps panel copy.** `AppsSettingsPanel.sections.test.ts` still expects `In workspace, not in my sidebar`, and `SettingsDialog.smoke.test.ts` still expects `My Sidebar`, while the panel now renders `Workspace app` and `Apps`/`Available`.
- **Tool-policy tests reference a missing implementation.** The full suite currently fails because `src/main/tools/toolPolicy.test.ts` imports `@main/tools/toolPolicy.js`, but no matching implementation exists. Finish or remove the in-progress tool-policy module and bring the related AI/runtime/server policy tests back into sync.
- **`mim list-tools` omits dynamic app tools.** The command creates a headless kernel but does not open a workspace before listing tools, so enabled named package tools are absent from the output even though `mim tool <named.app.tool>` works once a workspace is opened. Decide whether `list-tools --workspace <path>` should open the workspace or document that it is a static/core-only listing.
